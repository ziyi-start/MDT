# 🏥 医疗垂直领域多智能体协同 RAG 系统技术设计文档

**项目周期**：2026 年 5 月 - 至今
**文档版本**：v1.0
**核心技术栈**：LLM, SFT, Multi-Agent (ReAct), Milvus, Hybrid RAG, CoT, vLLM, Python

---

## 1. 项目概述

### 1.1 业务痛点
在传统医疗问答系统中，线性 RAG 架构面临三大核心挑战：
1.  **检索噪声大**：单向量检索缺乏精准度，常召回“表面语义相似但临床无关”的文献。
2.  **专业深度不足**：遇到跨科室并发症时，单模型无法覆盖多专科知识，回答笼统。
3.  **高危幻觉难控**：检索结果冲突或匮乏时，模型易强行生成（如编造药物相互作用），存在极大临床风险。

### 1.2 核心解法
本项目突破传统线性 RAG 局限，构建模拟临床 MDT（多学科会诊）的多智能体会诊框架。系统引入**闭环动态路由、渐进式画像约束检索与归因式反思沉淀**机制，实现从“被动检索生成”到具备主动决策与临床级安全纠偏能力的架构跃升。

## 3. 核心模块详细设计与实现

### 3.1 专家模型微调与高效部署
**目标**：让通用大模型具备“专科医生”的执业能力。

*   **数据清洗与单角色构造**：遍历 DISC-Med 数据集按轮次拆分。若对话涉及心血管，在 System Prompt 前缀注入 `[心内科专家]` 角色；过滤多科室混杂样本，保留单科室深度问答，构造标准 `{"instruction": "...", "input": "...", "output": "..."}` 格式。
*   **Llama-Factory 微调**：采用 QLoRA 算法（4-bit 量化 + 低秩适配），降低 13B 模型显存占用。关键参数：`lora_rank=64`, `lora_alpha=128`，目标模块涵盖 `q_proj, k_proj, v_proj, o_proj`。混入 10% 通用指令数据防止灾难性遗忘。
*   **vLLM 部署优化**：合并 LoRA 权重后，启动 vLLM 推理引擎。开启 PagedAttention 解决 KV Cache 显存碎片，支持 Continuous Batching，使多 Agent 并发请求吞吐量提升 2-3 倍。

### 3.2 闭环动态路由与置信度评估
**目标**：平衡系统效率与深度，实现 RAG 失败后的自动跃升。

*   **混合路由分流**：
    *   **规则拦截（快车道）**：使用医疗 NER 模型（BERT-CRF）提取实体。若 `实体数 <= 1` 且无冲突关键词（正则匹配），直接路由至 Simple RAG（耗时 <50ms）。
    *   **LLM 意图解析（慢车道）**：对灰度问题调用 Baichuan2，开启 Guided Decoding (JSON Constrained Generation)，强制输出路由 JSON（包含 `route_path` 和 `departments`）。
*   **置信度评估机制**：
    *   **文档一致性校验**：Reranker 打分后，若 Top-1 与 Top-2 分数差距 < 阈值且结论相悖，判定为“检索冲突”。
    *   **生成自验证**：要求 LLM 生成答案时输出引用来源 `[Source: Doc 1]`。若核心结论无对应文档引用，判定为“低置信度”。
*   **携因打回与动态升级**：触发低置信/冲突时，将失败原因（如“检索指南冲突，缺乏肾功能考量”）作为新 Context 注入 Recruiter，强制路由至 MDT 模式。

### 3.3 渐进式画像构建与共识引导检索
**目标**：解决表面语义相似但临床无关的误召回痛点。

*   **渐进式画像构建**：用户每轮对话后，异步触发轻量级 LLM 执行信息抽取（IE），将非结构化文本转化为结构化 JSON（`diseases`, `medications`, `allergies`）。以 `user_id` 为主键，通过 Milvus 的 Upsert 操作动态更新 `Patient_Profile` 集合。
*   **画像约束检索边界**：
    *   **硬约束**：若画像存在禁忌（如胃溃疡），在 Milvus 检索时增加元数据 Filter 条件，直接在数据库层排雷。
    *   **软约束**：在检索 Query 中拼接禁忌信息进行查询改写。
*   **多专家主动生成 Query (共识引导)**：Expert Agent 在 ReAct 循环中，主动调用 `Literature_Search` 工具生成检索词（如：肾内科专家主动生成“CKD 3期 痛风 安全止痛药替代方案”），取代用户原始 Query 进行检索。
*   **混合检索与重排**：基于 Milvus 2.3+ 的 Hybrid Search，同时执行 BM25（精准匹配药名）和 Dense（语义匹配），通过 RRF 算法合并结果。最终使用微调的医疗 BGE-Reranker (Cross-Encoder) 打压临床逻辑无关的噪声。

### 3.4 归因式反思沉淀与 CoT 安全退避
**目标**：应对极端噪声，建立系统级“免疫系统”，坚守临床安全底线。

*   **归因式反思沉淀**：
    *   **生成**：Decision-maker 打回结果后，触发归因 Prompt，强制输出结构化三元组：`<意图, 归因, 避坑动作>`（例：`<为CKD开止痛药, 忽略NSAIDs肾损风险, 必须核查肾功能禁用NSAIDs>`）。
    *   **写入与拦截**：将三元组向量化存入 Milvus `Reflection_Mem` 集合。下次遇到相似 Query 时，先检索反思记忆，若高相似度命中，则在专家 System Prompt 中强插 Hint 预警。
*   **CoT 安全退避机制**：
    *   **触发条件**：Milvus 检索结果的 Reranker 得分全部低于极低阈值（如 0.2），表明知识库无相关知识。
    *   **强制中断**：Decision-maker 切断 ReAct 循环，禁止发散推理。
    *   **退避执行**：走硬编码的保守策略链路，输出：“抱歉，缺乏权威文献支撑，强烈建议线下就医”，严禁编造具体用药方案（宁拒答不幻觉）。

## 4. 核心数据流转全生命周期（案例推演）

**场景**：*“患者有高血压和胃溃疡，最近痛风发作，能吃布洛芬吗？”*

1.  **路由阶段**：NER 提取出 `高血压`、`胃溃疡`、`痛风`、`布洛芬`。规则拦截器判定实体数>2，直接路由至 **MDT 模式**，招募 [心内科, 消化科, 风湿科]。
2.  **画像与检索阶段**：从 Milvus `Patient_Profile` 读出患者长期服用“氯吡格雷”；风湿科专家主动生成 Query：“胃溃疡 痛风 急性发作 非NSAIDs止痛方案”。
3.  **会诊与评估阶段**：风湿科建议用秋水仙碱，消化科提示仍需护胃。Decision-maker 评估共识无冲突，置信度达标。
4.  **反思/退避阶段（分支假设）**：若知识库完全没有关于该复杂并发症的用药记录，检索得分极低，系统将触发 **CoT 退避**，拒绝提供用药方案，输出安全就医建议；并将此次“无证据支持”的路径写入反思记忆。

## 6. 面试防坑与深度追问 Q&A

**Q1：你的画像/反思记忆是怎么生成的？不能是大模型随便生成的文本吧？**
**A**：绝对不是随便生成的。我们设计了结构化抽取管道。画像是通过异步轻量级 LLM 抽取 JSON 存入的；反思记忆是通过归因提示器，强制输出 `<意图,归因,避坑动作>` 三元组，确保经验是可被精确检索和作为规则执行的，而非模糊的自然语言。

**Q2：怎么判断生成的内容是低质/冲突的？置信度怎么算？**
**A**：双管齐下。一是“文档级校验”，Reranker 发现 Top 文档得分相近但结论相悖，判定冲突；二是“生成级自验证”，要求模型输出引用来源，如果关键结论没有文档强支撑，判定低置信度。这两者都会触发路由升级或反思打回。

**Q3：CoT 退避机制具体怎么执行？**
**A**：当检索得分低于极低阈值时，系统判定进入“信息匮乏边界”。此时系统会切断 Agent 的 ReAct 循环，不允许它继续思考或调用工具，而是走一条硬编码的保守策略链路，强制输出“信息不足+建议就医”。这在医疗场景是必需的生命线。

**Q4：动态路由怎么节省 Token 的？**
**A**：传统多智能体不管什么问题都启动全量 Agent。我们通过规则拦截（看实体数和关键词），把 60% 的单专科事实型问题挡在 MDT 之外，直接走单次 RAG。只有真正复杂的才启动多专家 ReAct 循环，这极大地节省了思考和工具调用的 Token 消耗。

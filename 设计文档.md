# 医疗 MDT 多智能体协同 RAG 系统 — 技术设计文档

**技术栈**: DeepSeek, Multi-Agent (ReAct), Milvus 2.5+, Hybrid RAG (BM25 + Dense), BGE-Reranker-v2-m3 Cross-Encoder, BGE-small-zh-v1.5 Embedding (512d), FastAPI | 语料: MedQA-US 英文教科书 18本 + 中文教科书 (段落版)

**新增框架**: (1) **Harness 工程** — `Agent = Model + Harness`，参考 OpenAI Harness Engineering; (2) **Agent 自进化** — 从成功回答提取可复用 Skills，add/merge/discard 版本管理; (3) **四库分层记忆** — Medical_KB / Patient_Profile / Reflection_Mem / Skill_Mem

## 2. 核心流程设计

### 2.1 动态路由（`router/`）

两级分流，由 `MedicalOrchestrator._route_async()` 编排：

**规则拦截（同步，<50ms）— `RuleInterceptor.intercept()`**

```python
entities = ner_extract(query)  # 正则或外部 BERT-CRF 服务
conflict_keywords = ["能...吗", "冲突", "相互作用", "同时服用"]
if len(entities) <= 1 and not has_conflict:
    return RouteDecision("simple_rag")
if len(entities) > 2 or has_conflict:
    return RouteDecision("mdt", departments=map_to_dept(entities))
# len(entities) == 2 且无冲突词 → 灰度区，返回 None 交 LLM 路由
```

科室映射表 `DEPARTMENT_MAP` 覆盖 20+ 常见医疗实体→科室（高血压→心内科，胃溃疡→消化科，痛风→风湿科等）。

**LLM 路由（异步，~1s）— `LLMRouter.route()`**

灰度问题调用 LLM，开启 `response_format={"type": "json_object"}` 强制输出结构化 JSON。解析失败时安全降级为 SimpleRAG。

### 2.2 Simple RAG（`workflow/simple_rag.py`）

SimpleRAG 内部流程：
```
反思拦截 → 检索 → 重排 → LLM 生成
```

置信度评估由外部 `MedicalOrchestrator` 在 SimpleRAG 返回后统一执行（参见 2.4 节准入控制）。

**反思拦截**: 先查 `Reflection_Mem`，命中则注入 `⚠️历史教训：{avoid_action}` 到 System Prompt。

**技能注入**: 检索 `Skill_Mem`，命中则追加 `[技能] {action}` 到 System Prompt（参见 6.3 节）。

**检索**: `HybridRetriever.retrieve(query, profile, top_k=10)`:
1. 软约束：将画像禁忌拼接到查询文本（`f"{query} 禁忌: {contraindications}"`）
2. 硬约束：构建 Milvus filter 排除含禁忌的文档
3. Milvus hybrid_search: Dense (BGE 512d, COSINE) + BM25 → RRF(k=60) 融合
4. 结果按 RRF 分降序，取 top_k

**重排**: `MedicalReranker.rerank(query, docs, top_k=5)`:
1. 主方案：BGE-Reranker-v2-m3 Cross-Encoder，以 `(query, doc.content)` pair 逐对打分，sigmoid 归一化到 [0,1]
2. 后备（模型加载失败时）：n-gram 中文分词重叠度评分
3. 低于 `low_threshold=0.2` → 触发 `InsufficientInformationException`（CoT 退避）

**LLM 生成**: 拼接 `参考文献：{context}\n\n问题：{query}`，要求输出 `[Source: Doc N]` 引用格式。

**置信度评估**: `ConfidenceChecker.evaluate()`:
- 文档一致性：Reranker Top-1 与 Top-2 分差 < 0.15 且结论相悖 → 冲突
- 生成自验证：LLM 检查核心结论是否有文档支撑
- 任一不通过 → 抛 `RouteEscalationException(reason)`，顶层捕获后升级到 MDT

### 2.3 MDT 多专家会诊（`workflow/mdt_consultation.py`）

```
反思拦截 → 并发专家 ReAct → 共识提炼 → 共识引导检索 → 证据验证 → 返回
```

**专家 ReAct 循环**（`engine/react_engine.py`）:

每个科室独立创建 `ReactEngine`，注入：
- `EXPERT_SYSTEM_PROMPT.format(department=科室名, reflection_hint=反思警告)` + `skill_hints`（若命中）
- 患者画像（疾病/用药/过敏）
- 如果是从 SimpleRAG 升级而来，携带 `escalation_reason`

ReAct 循环逻辑：
```
1. 调用 LLM，传入 system_prompt + 历史消息 + tools
2. 若 finish_reason == "tool_calls":
   - 解析 tool_name + arguments
   - 通过 ToolRegistry 执行对应异步函数
   - 将结果封装为 role:"tool" 的 message 追加到历史
   - 回到 step 1（最多 5 次）
3. 若 finish_reason == "stop": 返回最终文本
```

注册工具：
- `literature_search(query, departments, top_k)`：调用 HybridRetriever + Reranker，结果转文本
- `drug_interaction(drugs)`：从 Milvus 检索药物相互作用知识

**共识提炼**:
```
prompt = CONSENSUS_PROMPT.format(
    expert_opinions="[心内科]\n...\n\n[消化科]\n...",
    profile="疾病=高血压,胃溃疡 用药=氯吡格雷 过敏=无"
)
llm.chat(temperature=0.3)  →  结构化会诊报告
```

**共识引导检索（新增，关键环节）**:

```
consensus_text → HybridRetriever.retrieve(top_k=8)
              → MedicalReranker.rerank(top_k=4)
              → CONSENSUS_VERIFICATION_PROMPT.format(
                    consensus=共识文本,
                    evidence=检索证据,
                    profile=患者画像
                )
              → LLM 标记每条结论:
                   [Supported]  = 有文献支撑
                   [Revised]    = 与文献矛盾，已修正
                   [Removed]    = 无文献支撑，已删除
```

这一步强制将 LLM 生成的共识"接地"回知识库，防止专家凭经验发声却脱离文献。

**Decision Maker 评估**（`workflow/medical_orchestrator.py:DecisionMaker`）:

```python
prompt = DECISION_MAKER_PROMPT.format(report=共识报告, profile=患者画像)
# 强制 JSON 输出:
# {"quality_score": 0.8, "hallucination_risk": "low",
#  "logical_gaps": [], "safety_concerns": [],
#  "approved": true, "reason": ""}
```

评估后处理：
- `hallucination_risk == "high"` 或 `quality_score < 0.5` → CoT 退避 + 写反思
- 中等问题 → 输出回答，追加 `⚠️注意：{logical_gaps}`
- JSON 解析失败 → 默认通过（防误拦截）

### 2.4 安全退避（CoT Fallback）

触发场景：
1. Step 2 预检索返回空文档
2. Reranker 最高分 < 0.2（知识库无相关知识）
3. Decision Maker 检出高幻觉风险或极低质量

行为：顶层 `MedicalOrchestrator.process()` 捕获 `InsufficientInformationException`，切断一切 LLM 继续生成的可能，返回硬编码安全回复：

> "抱歉，缺乏权威文献支撑，强烈建议线下就医。"

触发场景 2/3 时，额外调用 `ReflectionManager.generate_and_store(query, reason)` 生成反思三元组 `<intent, cause, avoid_action>` 存入 Milvus Reflection_Mem，后续同类查询会被拦截注入 Hint。

成功回答则触发对称的技能提取流程（参见 6.1 节），由 `SkillManager.extract_from_success()` 提取可复用策略存入 Skill_Mem，实现正反双轨学习。

## 3. 模块详述

### 3.1 嵌入与向量库

**BGE-small-zh-v1.5**（`rag/embedding.py`）
- 512 维，L2 归一化
- 延迟加载：首次 `dummy_embed()` 调用时初始化 `SentenceTransformer`
- 结果缓存：`@lru_cache(maxsize=256)` 避免同一文本重复计算
- 种子知识库 8 条向量预计算为 `seed_vectors.json`，启动免加载模型

**Milvus 四集合**（`rag/milvus_client.py`）

| 集合 | 主键 | 字段 | 用途 |
|------|------|------|------|
| Medical_KB | doc_id | content, source, department, contraindications, embedding, sparse_embedding | 医学知识库，Dense+BM25 双索引 |
| Patient_Profile | user_id | diseases, medications, allergies, embedding | 患者画像，Dense 索引 |
| Reflection_Mem | triple_id | intent, cause, avoid_action, usage_count, embedding | 反思三元组（失败教训），Dense 索引 |
| Skill_Mem | skill_id | intent, action, departments, source_query, provenance, version, usage_count, last_used, status, embedding | 技能记忆（成功策略），Dense 索引 |

`create_kb_collection()` 自动比对配置的 `embedding_dim`，不匹配则删表重建。

### 3.2 检索器（`rag/hybrid_retriever.py`）

`HybridRetriever.retrieve()` 完整链路：

```python
# 1. 软约束：改写查询
rewritten = f"{query} 禁忌: {profile.contraindications}" if profile else query

# 2. 硬约束：构造 filter
filter_expr = " and ".join(
    f'not (contraindications like "%{c}%")' for c in profile.contraindications
)

# 3. Milvus hybrid_search
dense_req = AnnSearchRequest(data=[dense_vec], anns_field="embedding", ...)
sparse_req = AnnSearchRequest(data=[query_text], anns_field="sparse_embedding", param={"metric_type":"BM25"})
results = client.hybrid_search(reqs=[dense_req, sparse_req], ranker=RRFRanker(k=60))

# 4. 后过滤
filtered = [d for d in results if not profile_contraindicated(d, profile)]
return filtered[:top_k]
```

后备方案：Milvus 不可用时加载内存知识库 `_IN_MEMORY_KB`，使用中文 n-gram 重叠度评分。

### 3.3 重排器（`rag/reranker.py`）

```python
class MedicalReranker:
    def rerank(self, query, documents, top_k=5):
        try:
            return self._cross_encoder_rerank(query, documents, top_k)
        except Exception:
            return self._ngram_rerank(query, documents, top_k)

    def _cross_encoder_rerank(self, query, documents, top_k):
        model = CrossEncoder('BAAI/bge-reranker-v2-m3')
        pairs = [(query, doc.content) for doc in documents]
        scores = model.predict(pairs)        # raw logits
        float_scores = [1/(1+exp(-s)) for s in scores]

    def is_insufficient(self, documents):
        return max(d.score for d in documents) < self.low_threshold  # 0.2
```

### 3.4 画像与反思（`memory/`）

**`ProfileExtractor`**:
```python
# 异步 LLM 抽取，强制 JSON 输出
prompt = PROFILE_EXTRACTION_PROMPT.format(text=user_query)
# diseases=["高血压"], medications=["氯吡格雷"], allergies=[]
# upsert 到 Patient_Profile（Milvus upsert 按 user_id 覆盖）
```

**`ReflectionManager`**:
- `generate_and_store(query, reason)`: LLM 强制输出反思三元组 JSON → 编码 → 插入 Reflection_Mem
- `search_reflection(query)`: 检索 Reflection_Mem，返回最高分命中项的 avoid_action（超过 threshold=0.5），否则返回 None

### 3.5 ReAct 引擎（`engine/`）

**`ToolRegistry`**:
```python
@global_tool_registry.register(
    name="literature_search",
    description="检索医学文献...",
    parameters={...}
)
async def literature_search(query: str, top_k: int = 5) -> str:
    ...  # 调用 retriever.retrieve + reranker.rerank，返回格式化文本
```

**`ReactEngine.run(query)`**:
```python
messages = [Message(role="system", content=system_prompt)]
for i in range(max_iterations):
    resp = llm.chat(messages=messages, tools=tool_schemas)
    if resp.finish_reason == "stop":
        return resp.content
    # finish_reason == "tool_calls"
    for tc in resp.tool_calls:
        result = await tool_registry.execute(tc.name, tc.args)
        messages.append(ToolMessage(role="tool", content=result, tool_call_id=tc.id))
raise MaxIterationsError
```

## 4. 数据流（案例）

**查询**: "患者有高血压和胃溃疡，最近痛风发作，能吃布洛芬吗？"

```
Step 1 画像更新:
  NER/LLM 提取 → Patient_Profile: diseases=[高血压,胃溃疡,痛风]
  medications=[氯吡格雷] → upsert 到 Milvus

Step 2 预检索: Hybrid top_k=3，有结果

Step 3 路由:
  规则拦截: entities=[高血压,胃溃疡,布洛芬,痛风] ≥3 → MDT
  招募: [心内科, 消化科, 风湿科]

Step 4 MDT 会诊:
  ① 反思拦截: 检索 Reflection_Mem，无关 → 无 Hint
  ② 并发 ReAct (asyncio.gather):
     心内科专家: literature_search("高血压患者NSAIDs风险")
     消化科专家: literature_search("胃溃疡 止痛 安全方案")
     风湿科专家: literature_search("痛风急性发作 合并胃溃疡 用药")
  ③ 共识提炼: LLM 综合三专家意见 → 共识报告
  ④ 共识引导检索:
     retrieve(共识报告, top_k=8) → rerank(top_k=4) → LLM 验证
     [Supported] 禁用布洛芬（有文献支撑）
     [Revised]   备选方案增加PPI交互警告（证据[4]修正原共识）
     [Removed]   无
  ⑤ Decision Maker: quality_score=0.8, hallucination=low → 通过

Step 5 输出:
  路由: mdt, 科室: [心内科,消化科,风湿科]
  来源: 4 篇文献, 置信度: 1.0
  MDT 报告含 Supported/Revised 标记及具体文献编号
```

## 5. Harness 工程框架 (新增) — Agent 的工程外壳

### 设计理念

参考 OpenAI Harness Engineering、EleutherAI Agent Harness 综述、Anthropic Harness 设计:

```
Agent = 模型 + Harness
```

**核心公式**: Harness 是为模型提供操作环境的代码集合，包括工具、知识、观察接口、行动空间、约束规则和评估反馈。

**五大设计原则**:

| 原则 | 含义 | 本系统实现 |
|------|------|-----------|
| 上下文即预算 | 上下文像预算一样管理：永久层压缩≤2K，深度层按需加载 | `ContextManager` 三层预算 |
| 流程外化 | 流程约束从 LLM 推理中剥离到确定性基础设施 | `MedicalOrchestrator` 闭环 + `SafetyGuard` |
| 生成-评估分离 | 评估不使用 LLM，零 LLM 调用的确定性评分 | `HarnessEvaluator` 7维评分 |
| 可观测性优先 | 每个操作都有 trace，每个决策都有记录 | `TraceContext` Span + 执行图 |
| 实验驱动迭代 | 改完配置后必须通过评估才能判断好坏 | `ExperimentTracker` A/B 对比 |

### 5.1 观测层: 分布式追踪 (`monitoring/tracing.py`)

```
TraceContext (贯穿全流程)
  ├── Span: process              # 顶层处理
  │   ├── Span: profile_update   # 画像更新
  │   ├── Span: cot_precheck     # 预检索
  │   ├── Span: routing          # 动态路由 (含 route_path / departments)
  │   ├── Span: execute          # 执行阶段
  │   │   ├── Span: simple_rag   # Simple RAG 执行
  │   │   │   └── Span: confidence_check  # 置信度评估
  │   │   └── Span: mdt          # MDT 会诊
  │   │       └── Span: decision_maker    # Decision Maker 评估
  │   └── Span: [error/fallback] # 异常/退避
  └── [归档] → TraceManager (最近 1000 条)
```

**Span 属性**:
- `span_id`: 唯一标识
- `parent_id`: 父 Span ID (构建 DAG)
- `name`: 阶段名称
- `start_time / end_time`: 精确计时
- `duration_ms`: 耗时
- `status`: ok / error / fallback
- `metadata`: 阶段特定的元数据（如 route_path、num_docs）

**API 端点**:
- `GET /api/harness/traces` — 最近 N 条追踪
- `GET /api/harness/traces/{trace_id}` — 特定追踪详情（含完整 Span 树）
- `WS /ws/query` — WebSocket 实时查询端点，支持流式处理

### 5.2 评估层: 7维确定性评分 (`harness/evaluator.py`)

**设计决策**: 宁要可复现的"粗糙分"，不要会漂移的"精准分"。

**零 LLM 调用** — 所有评分基于确定性规则和已有指标，3 次跑分 hash 完全一致。

**七维评分体系**:

| 维度 | 权重 | 检查什么 | 怎么检查 |
|------|------|----------|----------|
| 流程完整性 | 22% | 该走的流程节点是否都走了？（路由、部门、来源、置信度） | 检查 response 中 route_path/departments/sources/confidence 字段 |
| 输出质量 | 15% | 回答是否有实质内容？有无空泛模板套话？ | 长度检查 + 空泛短语检测 + 来源引用 |
| 答案正确性 | 22% | 答案是否可靠？置信度是否合理？ | 平均置信度 × 0.6 + 退避惩罚 × 0.4 |
| 效率 | 10% | 端到端耗时？ | 平均延迟分段评分（<5s→1.0, 5-10s→0.5-1.0, >10s→惩罚） |
| 安全合规 | 8% | 安全退避是否正确触发？ | 退避时 confidence=0, 非退避时 confidence>0 |
| 迭代能力 | 5% | 失败后能否自我修复？ | SimpleRAG→MDT 升级率评分 |
| 接口验收 | 18% | 最终答案经得起验证？ | MDT 路径平均置信度 / SimpleRAG 路径平均置信度 |

**Regression 检测**:
```
compare(current, baseline) → {
  total_diff: -0.03,
  regressed_dimensions: { "quality": {baseline: 0.85, current: 0.72, diff: -0.13} },
  improved_dimensions: { "efficiency": {baseline: 0.60, current: 0.75, diff: 0.15} }
}
```

**API 端点**:

## 6. Agent 自进化 — 从成功中学习

### 设计理念

参考 Agent 自进化机制，MDT 实现了双轨学习：

| 轨道 | 信息来源 | 存储位置 | 核心操作 |
|------|----------|----------|----------|
| **失败反思** (已有) | 置信度不足/决策拦截时的失败回答 | `Reflection_Mem` | 提取反思三元组 → 存储 → 相似查询时注入 Hint |
| **成功技能** (新增) | 置信度达标的成功回答 | `Skill_Mem` | 提取规范策略 → add/merge/discard决策 → 注入上下文 |

### 6.1 成功技能提取流程 (`memory/skill_manager.py`)

```
成功回答 → LLM 提取技能 → 检索相似已有技能 → add/merge/discard 决策 → 存储
```

**提取条件**:
- 路由路径不为 safe_fallback / error
- 置信度 ≥ `min_confidence_for_extraction` (默认 0.7)
- 回答质量足够（LLM 判断）

**LLM 提取输出**:
```json
{
  "intent": "技能的触发意图（可泛化到同类场景）",
  "action": "规范化的处理策略（可操作、可复用）",
  "departments": ["适用科室1", "适用科室2"]
}
```

### 6.2 add/merge/discard 决策

检索到与现有技能相似度 ≥ `similarity_threshold` (0.75) 时，触发 LLM 决策：

| 决策 | 行为 | 版本变化 |
|------|------|----------|
| **add** | 保留新旧两条（覆盖不同子场景） | 旧版本不变 |
| **merge** | 合并新旧技能为新的优化版本 | 旧版本标记 superseded，新版本 version+1 |
| **discard** | 新技能质量低于旧版，丢弃 | 不存储 |

### 6.3 技能注入

每次查询时检索 `Skill_Mem`：

```
技能命中 → 格式化 skill_hints → 注入 System Prompt:
  参考既往成功经验：
  [技能] 高血压合并痛风止痛: 禁用NSAIDs，优先对乙酰氨基酚或秋水仙碱
```

技能提示 (skill_hints) 与反思提示 (reflection_hint) 互补：
- reflection_hint: "⚠️ 不要做 X"（失败教训）
- skill_hints:    "[技能] 应该做 Y"（成功经验）

### 6.4 Usage Stats 追踪

每次技能/反思被检索命中时递增 usage_count（"从经验中学习"的效果可衡量）。

**Skill 数据模型**:
```python
class Skill(BaseModel):
    skill_id: str          # 唯一标识
    intent: str            # 技能触发意图
    action: str            # 规范化处理策略
    departments: list[str] # 适用科室
    source_query: str      # 来源用户查询
    route_path: str        # 来源路由路径 (simple_rag / mdt)
    provenance: str        # 来源追踪标识
    version: int           # 版本号（merge 时递增）
    usage_count: int       # 被检索使用次数
    last_used: str         # 最后使用时间
    status: str            # active | superseded | discarded
```

### 6.5 配置

```yaml
skill:
  search_threshold: 0.6           # 技能检索相似度阈值
  similarity_threshold: 0.75      # 冲突检测阈值
  max_in_memory: 100              # 内存存储上限
  max_return: 3                   # 检索最多返回数
  extraction_enabled: true        # 是否启用技能提取
  min_confidence_for_extraction: 0.7  # 提取最低置信度
```

## 7. 四库分层记忆架构

MDT 的知识体系由四个独立但有机关联的集合构成：

```
Medical_KB (医学知识库)
  │  内容: 教科书、指南、文献片段
  │  类型: Dense + BM25 混合检索
  │  更新: 批量导入，不频繁变动
  │
  ├── Patient_Profile (患者画像)
  │    内容: 疾病、用药、过敏等结构化事实
  │    类型: Dense 向量检索
  │    作用: 个性化检索约束（硬约束+软约束）
  │
  ├── Reflection_Mem (反思记忆)  ← "知道什么不该做"
  │    内容: <意图, 归因, 避坑动作> 三元组
  │    来源: 失败回答 → LLM 归因提取
  │    作用: 同类查询注入 ⚠️ 警告
  │
  └── Skill_Mem (技能记忆)  ← "知道该怎么做"
       内容: <意图, 策略, 适用科室> 技能
       来源: 成功回答 → LLM 技能提取 + add/merge/discard
       作用: 同类查询注入 ✅ 最佳实践
```

**四库职责分离**:

| 集合 | 角色 | "知道什么" | "怎么做" | 来源 |
|------|------|-----------|---------|------|
| Medical_KB | 外部知识 | ✅ 领域事实 | ❌ | 教科书 |
| Patient_Profile | 个性化事实 | ✅ 患者信息 | ❌ | LLM 抽取 |
| Reflection_Mem | 内部经验 | ❌ | ✅ 反面教材 | 失败回答 |
| Skill_Mem | 内部经验 | ❌ | ✅ 正面范例 | 成功回答 |

Milvus 四集合管理:

```python
# rag/milvus_client.py
def init_all_collections(self):
    self.create_kb_collection()           # Medical_KB
    self.create_profile_collection()      # Patient_Profile
    self.create_reflection_collection()   # Reflection_Mem
    self.create_skill_collection()        # Skill_Mem (新增)
```

## 8. 监控与评估体系 (`monitoring/`)

### 8.1 运行时指标 (`metrics.py`)

**PipelineTimer** — 管道阶段计时上下文管理器：
```python
timer = PipelineTimer()
with timer.stage("retrieval"):
    docs = await retriever.retrieve(query)
with timer.stage("rerank"):
    reranked = await reranker.rerank(query, docs)
metrics = timer.to_metrics()  # RequestMetrics 实例
```

**RequestMetrics** 单次请求指标：

| 字段 | 类型 | 说明 |
|------|------|------|
| `query` | str | 用户查询文本 |
| `route_path` | str | 路由路径: simple_rag / mdt / safe_fallback |
| `total_latency_ms` | float | 端到端总延迟 |
| `retrieval_ms` | float | 检索阶段耗时 |
| `rerank_ms` | float | 重排序阶段耗时 |
| `generation_ms` | float | LLM 生成阶段耗时 |
| `confidence_check_ms` | float | 置信度校验耗时 |
| `retrieved_count` | int | 检索返回文档数 |
| `reranked_count` | int | 重排保留文档数 |
| `final_confidence` | float | 最终置信度 0-1 |
| `is_safe_fallback` | bool | 是否触发安全退避 |
| `top_retrieval_scores` | list[float] | 检索 Top-k 得分 |

**SessionMetrics** 会话聚合指标（`GET /api/metrics`）：

| 指标 | 说明 |
|------|------|
| `success_rate` | 请求成功率 |
| `safe_fallback_rate` | 安全退避触发频率 |
| `route_distribution` | 各路由路径调用次数分布 |
| `avg_total_latency_ms` | 平均总延迟 |
| `avg_retrieval_ms` | 平均检索延迟 |
| `avg_rerank_ms` | 平均重排延迟 |
| `avg_generation_ms` | 平均生成延迟 |
| `avg_confidence` | 平均置信度 |

### 8.2 离线评估 (`rag_evaluator.py` + `scripts/evaluate_rag.py`)

**检索质量指标** (`RetrievalEvaluator`):

| 指标 | 公式 | 说明 |
|------|------|------|
| MRR | Mean(1/rank₁) | 平均倒数排名，首个相关文档排名倒数均值 |
| Hit@1 | I(相关文档在Top-1) | Top-1 命中率 |
| Hit@3 | I(相关文档在Top-3) | Top-3 召回率 |
| Hit@5 | I(相关文档在Top-5) | Top-5 召回率 |
| Hit@10 | I(相关文档在Top-10) | Top-10 召回率 |

相关性判定：基于中文 2-gram 关键词重叠度（≥10% 视为相关）。

**生成质量指标** (`GenerationEvaluator`):

| 指标 | 计算方式 | 说明 |
|------|----------|------|
| Accuracy | 预测选项 == 标准答案 | 单选题答题准确率（直接/模糊/选项映射三重匹配） |
| Faithfulness | LLM-as-judge | 答案是否可从检索文档推导（防幻觉） |

**评测脚本** `scripts/evaluate_rag.py`:
```bash
# 生成准确率评测 (推荐)
python scripts/evaluate_rag.py --num 100 --top_k 5

# 检索召回率评测
python scripts/evaluate_rag.py --mode retrieval --top_k 10

# 全量评测 3,426 题
python scripts/evaluate_rag.py

# 同时评测检索+生成
python scripts/evaluate_rag.py --mode both
```
结果自动保存到 `data/eval_report_gen.json`。

### 8.3 已有安全/质量机制

| 组件 | 路径 | 功能 |
|------|------|------|
| `ConfidenceChecker` | `router/confidence_checker.py` | 文档一致性校验 (Top1-Top2 分差<0.15 且结论相悖→冲突) |
| | | LLM 生成自验证 (核心结论是否有文档支撑) |
| `DecisionMaker` | `workflow/medical_orchestrator.py` | quality_score + hallucination_risk + logical_gaps 评估 |
| `is_insufficient` | `rag/reranker.py` | Reranker 最高分 < 0.2 → CoT 安全退避 |
| 编排器日志 | `workflow/medical_orchestrator.py` | 每次查询输出: `延迟=XXms 召回=XXms 信心=X.XXX` |
